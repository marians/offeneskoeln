Die {{ config.APP_NAME }} API

API = engl. Application Program Interface: Eine Programmierschnittstelle für den Zugriff auf ein System.

Das Ziel von {{config.APP_NAME}} ist es, Daten und Informationen zur {{config.CITY_NAME_ER}} Lokalpolitik so weit wie möglich für jede denkbare Nutzung zu öffnen. Dazu gehört auch die Öffnung für den Zugriff mit Hilfe von Programmen und Systemen.

{{ config.BASE_HOST }} bietet einige Zugriffsmöglichkeiten auf den Datenbestand als Webservice-API. Genau wie die Plattform selbst ist die API noch in einem sehr, sehr frühen Reifestadium und wird mit großer Wahrscheinlichkeit noch vielen Änderungen unterworfen sein.

Dokumentation

Ausgabeformate

Die API unterstützt die Ausgabe in den Formaten JSON und JSONP. Ist der URL-Parameter callback gesetzt, wird JSONP ausgegeben, sonst JSON.

HTTP-Methode

Sämtliche Anfragen an die API erfolgen mit der HTTP-Methode GET.

Kompression

Der Server unterstützt GZip-Kompression der Antwort. Dafür muss der Client den String gzip im HTTP Request-Header Accept-Encoding senden.

Zeichkodierung von URL-Parametern

Alle URL-Parameter müssen mit Zeichensatz UTF-8 sowie URL-kodiert (RFC 3986) übergeben werden. Beispiel: Der Straßenname Hauptsstraße wird in einer URL zu Hauptsstra%C3%9Fe, wobei das Zeichen ß durch die zwei aufeinander folgenden hexadezimal-Zeichen %C3 und %9F kodiert wird.

Geodaten

Geo-Koordinaten werden aus einer Kombination aus Längen- und Breitengradangabe im WGS84-System wiedergegeben. Das bedeutet, die Angaben sind kompatibel mit GPS, Google Maps und OpenStreetMap.

Wenn eine Methode ein Koordinatenpaar ausgibt, ohne die beiden Werte einzeln zu benennen, steht immer als erstes der Breitengrad (lat bzw. latitude) und als zweites der Längengrad (lon bzw. longitude). Gradangaben sind Fließkommazahlen wie z.B. 6.496.

Methoden der API

Nachfolgend werden die einzelnen Methoden/Endpunkte beschrieben:

  • /api/documents - Abruf von Dokumenten einschl. Suche
  • /api/locations - Ausgabe von Geo-Positionen zu Straßen
  • /api/streets - Ausgabe von Straßen zu Geo-Positionen
  • /api/session - Lesen und Schreiben von Session-Daten
  • /api/terms - Auslesen von Dokumenteneigenschaften nach Feldern
  • /api/proxy/geocode - Nichtöffentliche Proxy-Methode

/api/documents

Mit dieser Methode können einzelne oder mehrere Dokumente anhand ihrer Kennung abgerufen oder anhand von weiteren Kriterien gesucht werden. Dokumente sind Beschlussvorlagen, Anträge und andere Dokumente, die im System erfasst sind, mitsamt den dazugehörigen Anhängen (PDF-Dateien etc.).

Modus-Unterscheidung zwischen Suche und Direktabruf

Bei dieser Methode gibt es zwei unterschiedliche Modi:

  • Direktabruf: ein Dokument (oder mehrere Dokumente) wird anhand einer Kennung (reference) bzw. einer Liste von Kennungen abgefragt.
  • Suche: Ist der URL-Parameter reference nicht gesetzt, handelt es sich um den Modus Suche.

Der URL-Parameter reference kann nur für den Direktabruf genutzt werden, ist also nicht mit Parametern der Suche kombinierbar. Einige andere URL-Parameter können nur im Suchmodus verwendet werden.

Beispielhafte »Direktabrufe«:

  • /api/documents?reference={{ config.API_DOC_EXAMPLE_REFERENCE }}
    Direktabruf des Dokuments mit der Kennung {{ config.API_DOC_EXAMPLE_REFERENCE }}.
  • /api/documents?reference={{ config.API_DOC_EXAMPLE_REFERENCE }}&output=attachments
    Wie zuvor, zusätzlich werden Details zu Anhängen ausgegeben.
  • /api/documents?reference={{ config.API_DOC_EXAMPLE_REFERENCE }}&output=attachments,thumbnails
    Wie zuvor, zusätzlich werden Daten zu Vorschaubildern ausgegeben.
  • /api/documents?reference={{ config.API_DOC_EXAMPLE_REFERENCE }}&output=consultations
    Direktabruf, zusätzlich werden Details zu den Sitzungen, in denen das Dokument beraten wurde/wird, ausgegeben.

Beispielhafte »Suchabfragen«:

  • /api/documents
    Suche ohne Suchabfrage oder Filter. Ruft die 10 neusten Dokumente ab.
  • /api/documents?q=fahrrad
    Suche nach Dokumenten, in denen der Begriff »Fahrrad« vorkommt. Es werden auch ähnliche Schreibweisen gefunden.
  • /api/documents?fq=type:Antrag
    Suche nach Dokumenten vom Typ »Antrag«

Parameter:

Alle Parameter sind optional.

reference

Die eindeutige Kennung eines Dokuments, z.B. {{ config.API_DOC_EXAMPLE_REFERENCE }}. Wie alle Angaben, die in den URLs übergeben werden, muss auch dieser Wert URL-kodiert sein.

Mehrere Kennungen können getrennt durch Komma übergeben werden, entsprechend werden mehrere Dokumente ausgegeben.

Modus: Wenn der reference-Parameter gesetzt ist, handelt es sich bei der Abfrage um einen Direktabruf. Auf die Suche bezogene Parameter wie q, fq, date, sort und docs können in diesem Modus nicht verwendet werden.

output

Der Parameter bestimmt, welche Details in den ausgelieferten Dokumentendaten enthalten sein sollen. Je mehr Details aktiviert werden, desto länger dauert die Bearbeitung der Abfrage und desto umfangreicher werden die zurückgelieferten Daten.

Es können die Werte attachments, thumbnails, consultations und facets verwendet werden. Mehrere Werte können, durch Komma getrennt, mit einander kombiniert werden.

attachments aktiviert die Ausgabe von Daten zu Anhängen, die mit einem Dokument verknüpft sind. Dazu gehören im gegebenen Fall auch Information zur Sperrung bzw. Depublikation eines Anhangs (Details zu Depublikation im Wiki).

thumbnails aktiviert die Ausgabe von Daten zu allen Vorschaubildern zu den Anhängen des Dokuments. Damit dies funktioniert, muss auch attachments angegeben werden.

consultations aktiviert die Ausgabe der Beratungsfolge eines Dokuments, also die Liste der Sitzungen, in der ein Dokument vorgelegt wurde.

facets aktiviert die Ausgabe von Facetten, was nur im Modus Suche möglich ist. Bei einer Direktabfrage hat dieser Wert keine Wirkung. Facetten sind eine Zusammenfassung darüber, welche Dokumenteneigenschaften im Suchergebnis wie häufig vorkommen. Sie können zum Beispiel dafür benutzt werden, um Suchen Optionen für die weitere Einschränkung des Suchergebnisses vorzuschlagen.

Modus: Der Parameter output kann grundsätzlich unabhängig vom Modus (Suche oder Direktabruf) eingesetzt werden. AUSNAHME: Der Wert facets ist nur im Modus Suche wirksam.

q

Dieser Parameter bestimmt eine Suchabfrage, der Name q steht für »Query«. Ist dieser Parameter leer oder nicht gesetzt, entspricht dies einer Wildcard-Suche nach allen Dokumenten.

Suchabfragen, die hier übergeben werden, werden flexibel interpretiert. Hier werden zum Beispiel Synonyme berücksichtigt (Ampel, Lichtsignalanlage und LSA sind Synonyme). Auch werden verschiedene Wortformen auf den Stamm reduziert (Fußbälle wird zu Fußball).

Mehrere Suchbegriffe werden durch Leerzeichen (%20 bzw. +) getrennt. Je mehr Suchbegriffe kombiniert werden, desto weniger Treffer werden generell im Ergebnis enthalten sein. Es werden nur Dokumente zurückgegeben, die allen Suchbegriffen entsprechen.

Suchabfragen können Lucene Query Syntax enthalten. Das bedeutet, es können komplexe Suchabfragen mit Klammerung und Operatoren wie + und -, AND, OR und NOT gebildet werden. Durch Kombination mit dem Parameter fq (siehe weiter unten) kann die Suche weiter eingeschränkt werden.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar. Das heißt, er kann nicht mit dem Parameter reference kombiniert werden.

fq

Einschränkung des Suchergebnisses durch strikte Filter (fq steht für »Filter Query«).

Beispiel: Es sollen nur Dokumente vom Typ »Antrag« gefunden werden. Hierzu wird der URL-Parameter fq=type:Antrag in die Abfrage eingefügt.

Im einfachsten Fall ist das Schema für solche Filter auf Basis von Dokumenteneigenschaften eigenschaft:wert. Im genannten Beispiel ist type der Name der Eigenschaft und Antrag der Wert. Der Name der Eigenschaft wird vom Wert mit Doppelpunkt (: bzw. %3A) getrennt. Werte, die Leerzeichen enthalten, müssen in Anführungszeichen (" bzw. %22) gesetzt werden. Beispiel: fq=committee_name%3A%22{{ config.API_DOC_EXAMPLE_COMMITTEE_NAME|urlencode }}%22

Komplexe Filter Queries können durch Verwendung von Lucene Query Syntax formuliert werden.

Beispiel: type:(Antrag OR Beschlussvorlage) AND committee_name:Rat

URL-kodiert: fq={{ "type:(Antrag OR Beschlussvorlage) AND committee_name:Rat"|urlencode }}

Die Verwendung von Filter Queries (Parameter fq) zur Einschränkung des Suchergebnisses ist effizienter und damit schneller als die Verwendung von Queries (Parameter q). Das liegt zum einen daran, dass die mittels fq definierten Filter weniger flexibel sind. Es findet zum Beispiel keine Fehlertolerante Interpretation der Angaben statt oder Anwendung von Synonymen statt. Außerdem werden die mittels Query Filter erzeugten Suchergebnisse auf dem Server gecached, was das erneute Auffinden von Dokumenten mit der selben Einschränkung extrem beschleunigt. Wenn immer möglich, sollten daher Query Filter anstelle von Queries verwendet werden.

Die zur Auswahl stehenden Dokumenteneigenschaften für derartige Filter sind:

Eigenschaft Beschreibung
title Dokumententitel
term Stichworte
type Art des Dokuments
person Person
attachment Anhang
session Sitzung
committee_name Gremium
street Straße

Im Parameter fq können mehrere Filter mit einander kombiniert werden. Wie beim Parameter q kann ein Leerzeichen benutzt werden, um eine implizite UND-Verknüpfung zu erreichen. Um andere logische Verknüpfungen zu bewirken, stehen die boolschen Operatoren OR und NOT zur Auswahl.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar. Er kann also nicht eingesetzt werden, wenn der Parameter reference gesetzt ist.

Weitere Möglichkeiten, die Treffermenge einzuschränken, bestehen über die Parameter q und date.

docs

Anzahl Treffer-Dokumente, die zurück gegeben werden sollen. Standard ist 10, Maximum ist 100. Mit dem Wert 0 kann die Dokumentenausgabe unterdrückt werden. Je weniger Dokumente ausgegeben werden, desto schneller ist die Suche und desto weniger Daten werden übertragen. Dies kann nützlich sein, wenn nur die Anzahl der Treffer benötigt wird, nicht aber die Treffer selbst.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar.

start

Anzahl der Dokumente, die am Anfang des Suchergebnisses ausgelassen werden sollen. Damit kann durch ein Suchergebnis »geblättert« werden. Ist der Parameter nicht gesetzt, entspricht dies start=0.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar.

sort

Bestimmt die Sortierung des Suchergebnisses. Wenn nicht gesetzt, wird der Standard in Abhängigkeit davon gesetzt, ob der Parameter q gesetzt ist oder nicht. Ist q gesetzt, dann ist die Standard-Sortierung score desc, was bedeutet »Sortierung nach Relevanz, absteigend». Ist q nicht gesetzt, ist die Sortierung standardmäßig date desc (nach Datum, neuste Dokumente zuerst). Die dritte mögliche Sortierung ist date asc (nach Datum, älteste Dokumente zuerst).

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar.

date

Erlaubt die Einschränkung des Suchergebnisses auf Dokumente, die in einem bestimmten Datumsbereich veröffentlicht wurden.

Das Format entspricht [<Anfangsdatum>][-][<Enddatum>], wobei alle drei Teile optional sind. Für die Angabe des Anfangsdatums und des Enddatums kann jeweils eine von drei Genauigkeiten gewählt werden: Jahr, Monat oder Tag. Dies geschieht durch Angabe des jeweiligen Datums in einem der drei Formate YYYY, YYYYMM oder YYYYMMDD (Y-Jahr, M=Monat, D=Tag).

Wird nur ein Datum ohne vorangestelltes oder nachfolgendes - Zeichen angegeben, dann werden alle Dokumente ausgegeben, deren Datum mit dem angegebenen Datum übereinstimmt. Zum Beispiel werden bei date=2010 alle Dokumente aus dem Jahr 2010 getroffen und bei date=201011 alle Dokumente aus November 2010.

Wird nur ein Anfangsdatum mit anschließendem - Zeichen angegeben, bedeutet dies, dass ein nach oben offener Bereich definiert wurde. Beispielsweise findet die Angabe date=2010- alle Dokumente, die im Jahr 2010 oder später veröffentlicht wurden. date=20110315- findet alle Dokumente, die ab dem 15. März 2011 veröffentlicht wurden.

Umgekehrt bedeutet ein vorangestelltes - Zeichen, dass der Datumsbereich nach unten (in die Vergangenheit) unbegrenzt ist. date=-20110315 findet also alle Dokumente, die bis einschließlich zum 15. März 2011 veröffentlicht wurden.

Mit einem Anfangs- und Enddatum lässt sich schließlich ein exakt begrenzter Datumsbereich definieren. date=2010-2012 steht beispielsweise für den Zeitraum vom 1. Januar 2010 bis zum 31. Dezember 2012. date=201201-201202 steht für die Zeit vom 1. Januar bis 29. Februar 2012 (ja, Schaltjahre werden berücksichtigt) und date=20120204-20120206 für die Zeit vom 4. bis zum 6 Februar 2012.

Anfangs- und Enddatum müssen nicht die selbe Genauigkeit haben. Beispielsweise ist date=2010-20120206 eine gültige Angabe.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar.

lastmod_date

Datum und Zeit der letzten Änderung des Dokuments. Funktioniert ansonsten wie date.

Modus: Dieser Parameter ist nur im Modus Suche einsetzbar.

response → facets

Die Facetten liefern die Information, welche Dokumenteneigenschaften im Suchergebnis mit welcher Häufigkeit auftreten. Sie können für die weitere Eingrenzung des Suchergebnisses verwendet werden. Zu jeder Facette gibt es Wertepaare aus einer Zeichenkette und einer Zahl, wobei die Zahl die Anzahl der Treffer repräsentiert. Die Ausgabe ist auf 30 Elemente je Facette begrenzt, hierbei werden jeweils die Werte mit der höchsten Trefferzahl ausgegeben.

Zeigt an, welcher Wert der Dokumenten-Eigenschaft typ wie häufig im aktuellen Suchergebnis vorkommt.

Zeigt an, welche Werte der Dokumenten-Eigenschaft gremium wie häufig im aktuellen Suchergebnis vorkommen. Die vorangestellte Zahl beim Wert ist die numerische ID des Gremiums.

Zeigt an, welcher Wert der Dokumenten-Eigenschaft person wie häufig im aktuellen Suchergebnis vorkommt. Die vorangestellte Zahl beim Wert ist die numerische ID der Person.

Zeigt an, welche Werte der Dokumenten-Eigenschaft schlagwort wie häufig im aktuellen Suchergebnis vorkommen. Die Eigenschaft schlagwort wird aus dem gesamten Volltext des jeweiligen Dokuments gebildet, wobei alle Schreibweisen auf Kleinschreibung reduziert und eine Vielzahl von Wörtern herausgefiltert werden.

Ein weiterer Unterschied ist, dass hier auch Null-Werte ausgegeben werden und anzahl der Werte nicht limitiert ist, sondern immer den Zeitraum ab Oktober 2006 abdeckt, also seit der Veröffentlichung der ältesten Dokumente im Bestand.

/api/locations

Gibt alle Geo-Positionen aus, die zum angegebenen Straßennamen im {{config.CITY_NAME_ER}} Stadtgebiet vorliegen.

Datenherkunft und Lizenz

Die Koordinatendaten der einzelnen Geo-Posiotionen stammen vom OpenStreetMap Projekt und unterliegen entsprechend der Lizenz CC-BY-SA (siehe dazu Daten).

Beispielhafte Abfragen:

  • /api/locations?street=Roncalliplatz
    Gibt alle Details zum »Roncalliplatz« aus.
  • /api/locations?street=Marzellenstra%C3%9Fe&output=averages
    Gibt die Durchschnitts-Position für die Marzellenstraße aus.
  • /api/locations?street=Aachener%20Stra%C3%9Fe&output=nodes
    Gibt die einzelnen Knoten der Aachaner Straße(n) aus.

Parameter:

street (Pflicht)

Name der Straße oder des Platzes, dessen Position(en) ausgegeben werden soll.

output

Bestimmt, welche Details zu der angegebenen Straße ausgegeben werden. Kann die Werte nodes und averages enthalten, sowie eine mit Komma getrennte Kombination aus beidem. Default-Wert ist nodes,averages

nodes aktiviert die Ausgabe der einzelnen Positionen aller Knoten, die bei OpenStreetMap zu der/den Straße/n mit dem angegebenen Namen vorhanden sind.

averages aktiviert die Ausgabe von Durschschnitts-Koordinaten. Diese können beispielsweise verwendet werden, um die gesuchte Straße mit einem Marker auf einer Karte zu markieren. Dabei ist zu beachten, dass bisher nicht betrachtet wird, ob es sich bei den Einzelkoordinaten um dieselbe Straße oder um verschiedene Straßen handelt. Daher kann es vorkommen, dass die Durchschnittsposition aller Knoten irgendwo zwischen zwei oder mehr Straßen gleichnamigen Straßen liegt. Siehe hierzu auch Issue #21

Beispielhafte Ausgabe:

/api/locations?street=Alpenerplatz:

/api/streets

Dient zum Auffinden von Straßen im Umkreis einer Geo-Position im {{config.CITY_NAME_ER}} Stadtgebiet. Die Ausgabe erfolgt aufsteigend sortiert nach Entfernung zum Ausgangspunkt.

In den Geodaten des OpenStreetmap Projekts besteht jede Straße aus mehreren Punkten. Damit sich eine Straße als im Umkreis zu einer Position befindlich qualifiziert, muss mindestens einer der Punkte der Straße in dem Umkreis liegen.

Datenherkunft und Lizenz

Die Straßennamen entstammen den Daten des OpenStreetMap Projekts und unterliegen entsprechend der Lizenz CC-BY-SA (siehe dazu Daten).

Beispielhafte Abfrage:

  • /api/streets?lat=50.959&lon=6.946
    Straßen im Umkreis der Position 50,959/6,946 mit Radius 500 Meter

Parameter:

lat (Pflicht)

Breitengrad der Geo-Position, um die herum Straßen gefunden werden sollen. Dezimalschreibweise.

lon (Pflicht)

Längengrad der Geo-Position, um die herum Straßen gefunden werden sollen. Dezimalschreibweise.

radius

Radius des Umkreises in Metern. Standard: 500. Kann maximal 1000 Meter betragen.

Beispielhafte Ausgabe:

/api/streets?lat=50.959&lon=6.946 (gekürzt):

/api/session

Lesen und Schreiben von Session-Eigenschaften.

Wenn die Geo-Position des Nutzers (mittels Parameter lat und lon) gesetzt wird, wird automatisch versucht, die Adresse zu dieser Position, sofern sie im {{config.CITY_NAME_ER}} Stadtgebiet liegt, auzulösen. Dies geschieht im Hintergrund über die Yahoo Placefinder API.

Beispielhafte Abfragen:

  • /api/session
    Gibt den vollständigen Inhalt des Session-Objekts aus.
  • /api/session?lat=50.959&lon=6.946
    Setzt die Geo-Position des Nutzers auf 50.959,6.946.
  • /api/session?location_entry=Aachener%20Strasse%20(50674)
    Setzt die zuletzt vom Nutzer eingegebene Position auf »Aachener Strasse (50674)«

Parameter

Alle Parameter sind optional.

lat

Breitengrad der Geo-Position des Nutzers, die gesetzt werden soll. Dezimalschreibweise.

lon

Längengrad der Geo-Position des Nutzers, die gesetzt werden soll. Dezimalschreibweise.

location_entry

Die letzte vom Nutzer angegebene Positionsbeschreibung bzw. Adresse.

Beispielhafte Ausgabe:

/api/session:

/api/terms

Dient zur Ausgabe von Stichwörtern (Dokumenten-Eigenschaft term) mit einem bestimmten Präfix, z.B. für Auto-Vervollständigung. Wird der Parameter prefix nicht übergeben, werden die häufigsten Schlagworte zurückgegeben.

Beispielhafte Abfragen:

  • /api/terms?prefix=schau
    Gibt die Top 10 Stichwörter aus, die mit »schau« anfangen
  • /api/terms
    Gibt die Top 10 Stichwörter aus

Diese Funktion wird auch bei {{config.BASE_HOST}} aktuell noch nicht eingesetzt und ist daher noch als experimentell zu verstehen. Sie wird ausführlicher dokumentiert, sobald sie in Produktion geht.

/api/proxy/geocode

Die Methoden unter /api/proxy sind nicht zur öffentlichen Nutzung gedacht.

Bei der Methode /api/proxy/geocode handelt es sich um eine Proxy-Methode für die Yahoo! Placefinder API, mit deren Hilfe Ortsangaben wie z.B. Adressen in Geo-Positionen aufgelöst werden können. Wenn Du die Placefinder API nutzen möchtest, geh bitte direkt auf die entsprechende Website bei Yahoo und richte Dir dort einen eigenen App-Key ein. Das geht einfach und kostet nichts.
